# Настройки уведомлений

ADOS отправляет уведомления о прогрессе и результатах через Slack, Discord, Email или Signal.

## Поддерживаемые каналы

| Канал | Настройка |
|-------|-----------|
| **Slack** | Webhook URL |
| **Discord** | Webhook URL |
| **Email** | Адрес электронной почты |
| **Signal** | Привязка по QR-коду (через signal-cli-rest-api) |

## Интеграция со Slack

### 1. Создание Incoming Webhook

1. Создайте новое приложение на [Slack API](https://api.slack.com/apps)
2. Включите **Incoming Webhooks**
3. Нажмите **Add New Webhook to Workspace** и выберите канал
4. Скопируйте Webhook URL

### 2. Регистрация в ADOS

1. Перейдите в **Settings** → **Notifications** в панели управления
2. Выберите вкладку **Slack**
3. Вставьте Webhook URL
4. Нажмите **Send Test Notification** для проверки
5. Нажмите **Save**

### Примеры уведомлений

```
🚀 ADOS | Обработка Issue #42
Репозиторий: my-org/my-repo
Заголовок: Add rate limiting to the login API
Агент: Claude (claude-sonnet-4)
Статус: Реализация...
```

```
✅ ADOS | PR создан
Репозиторий: my-org/my-repo
PR #123: Add rate limiting to the login API
Изменённые файлы: 3 файла (+45 -12)
Ревью → https://github.com/my-org/my-repo/pull/123
```

## Интеграция с Discord

### 1. Создание Webhook

1. Настройки сервера Discord → **Интеграции** → **Webhook'и**
2. Нажмите **Новый Webhook**
3. Выберите канал
4. Нажмите **Скопировать URL Webhook'а**

### 2. Регистрация в ADOS

1. Перейдите в **Settings** → **Notifications** в панели управления
2. Выберите вкладку **Discord**
3. Вставьте Webhook URL
4. Нажмите **Save**

## Email-уведомления

1. Перейдите в **Settings** → **Notifications** в панели управления
2. Выберите вкладку **Email**
3. Укажите адрес для уведомлений
4. Нажмите **Save**

## Интеграция с Signal

Уведомления Signal используют [signal-cli-rest-api](https://github.com/bbernhard/signal-cli-rest-api) в качестве Docker-сайдкара на оркестраторе. Вместо ручной настройки URL API и номеров телефона ADOS предоставляет процесс привязки по QR-коду в один клик.

### Настройка

1. Перейдите в **Settings** --> **Notifications** в панели управления
2. Выберите вкладку **Signal**
3. Выберите оркестратор для размещения Signal-контейнера
4. Нажмите **Start Container** — ADOS запускает контейнер signal-cli-rest-api на выбранном оркестраторе
5. На экране появится QR-код. Отсканируйте его приложением Signal на телефоне для привязки устройства
6. После привязки статус изменится на **Connected**
7. Нажмите **Send Test Notification** для проверки
8. Нажмите **Save**

### Архитектура

- signal-cli-rest-api работает во внутренней Docker-сети (`ados-signal-net`) без привязки к портам хоста
- Путь коммуникации: Браузер --> Cloud Run API --> Очередь команд Firestore --> Опрос оркестратора --> Контейнер signal-cli
- Постоянный том (`ados-signal-data`) сохраняет данные привязки между перезапусками контейнера
- Оповещение о старом контейнере (`EventSignalContainerStale`) срабатывает, когда контейнеру более 60 дней

### API-эндпоинты

| Эндпоинт | Метод | Описание |
|----------|-------|----------|
| `/api/v1/signal/qrcode` | GET | Получить QR-код для привязки устройства |
| `/api/v1/signal/status` | GET | Получить статус Signal-контейнера |
| `/api/v1/signal/start` | POST | Запустить Signal-контейнер |
| `/api/v1/signal/stop` | POST | Остановить Signal-контейнер |
| `/api/v1/signal/send-test` | POST | Отправить тестовое уведомление |

## Уведомления об инцидентах

Помимо уведомлений о прогрессе задач, ADOS может отправлять одноразовые уведомления об инцидентах при критических сбоях системы. Они предназначены для операционного оповещения, а не для обновлений по каждой задаче.

### Поддерживаемые типы инцидентов

| Событие | Описание |
|---------|----------|
| **GitHub OAuth-токен истёк** | Не удалось обновить токен |
| **Claude MAX-токен истёк** | Ошибка `invalid_grant` при обновлении |
| **Оркестратор офлайн** | Оркестратор прекратил отправку отчётов |
| **Ошибка группы Runner** | Сбой согласования группы Runner |

### Каналы

Уведомления об инцидентах можно отправлять через:

- **Email** — через SMTP или Firebase Firestore mail collection
- **Webhook** — HTTP POST (совместим со Slack / Discord)
- **Signal** — через signal-cli-rest-api

### Дедупликация

- Каждый инцидент уведомляется только один раз, пока не будет разрешён (без спама)
- При устранении проблемы (например, успешное обновление токена) инцидент автоматически сбрасывается, что позволяет повторное уведомление при рецидиве

### Настройка

1. Перейдите в **Settings** --> **Incident Notifications** в панели управления
2. Переключайте отдельные типы событий
3. Настройте хотя бы один канал (Email, Webhook или Signal)
4. Нажмите **Send Test Notification** для проверки
5. Нажмите **Save**

### Переменные окружения (Email через SMTP)

| Переменная | Описание |
|------------|----------|
| `SMTP_HOST` | Имя хоста SMTP-сервера |
| `SMTP_PORT` | Порт SMTP-сервера |
| `SMTP_USERNAME` | Имя пользователя SMTP |
| `SMTP_PASSWORD` | Пароль SMTP |
| `SMTP_FROM` | Адрес отправителя |

> [!TIP]
> Если расширение Firebase «Trigger Email from Firestore» настроено, ADOS использует его как основной способ отправки email и переключается на прямой SMTP только при сбое записи в Firestore.

## События уведомлений

Настройте, какие события вызывают уведомления:

| Событие | По умолчанию | Описание |
|---------|-------------|----------|
| Начало обработки задачи | ✅ | ИИ начал обработку задачи |
| Создан PR | ✅ | PR создан |
| CI успешно | ❌ | CI/CD завершился успешно |
| CI сбой | ✅ | CI/CD завершился сбоем |
| Авторемонт | ✅ | Pipeline Watcher выполнил ремонт |
| SRE-оповещение | ✅ | SRE-агент обнаружил проблему |
| Бэклог сгенерирован | ❌ | AI-генерация бэклога завершена |
| Ошибка | ✅ | Ошибка при обработке |

## Контроль частоты уведомлений

Предотвращение перегрузки каналов уведомлениями:

- **Мгновенные уведомления**: Критические события (ошибки, SRE-оповещения)
- **Пакетные уведомления**: Ежедневный дайджест
- **Без уведомлений**: Отключение для определённых событий
